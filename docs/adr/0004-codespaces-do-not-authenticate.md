# 0004 — Codespaces do not authenticate; live runs go through CI

**Status:** Superseded by [ADR-0012](0012-codespace-authenticates-locally.md)
**Date:** 2026-05-14

## Context

[ADR-0003](0003-workload-identity-federation-for-ci-auth.md) commits CI to
keyless auth via Workload Identity Federation. The natural follow-on
question is: what does an interactive run from a Codespace or laptop use?

Three options:

1. **Codespace uses ADC + service-account impersonation.** Developer runs
   `gcloud auth application-default login`, gets `roles/iam.serviceAccountTokenCreator`
   on the SA, and writes an `impersonated_service_account` style
   `oauth2service.json`. No key on disk.
2. **Codespace stores a short-lived SA key as a Codespaces secret.** Key
   gets written to `~/.gam/oauth2service.json` at session start, rotated
   periodically.
3. **Codespace never authenticates.** All live runs go through CI; the
   codespace only edits code and tests against fixtures.

Investigation showed option 1 doesn't work: GAM7 has its own credential-file
loader (not generic `google.auth.default()`) and only recognizes two
shapes — classic `service_account` files and its own `external_account`
variant generated by gcloud's `create-cred-config`. It does not parse
`impersonated_service_account` files. So ADC + impersonation looks right on
paper but GAM never engages the SA path.

Option 2 is what Google's auth decision tree calls out as the case that
*does* warrant a key — when the platform genuinely can't do WIF. But it
contradicts the spirit of [ADR-0003](0003-workload-identity-federation-for-ci-auth.md)
and introduces a key-handling story (rotation, secret hygiene) that adds
operational surface for a developer convenience.

Option 3 has a cost — slower iteration on changes that touch real
Workspace behaviour — but the audit logic itself is testable against
fixtures, and the only thing CI proves that fixtures don't is whether
the auth chain is wired up.

## Decision

Codespaces (and dev laptops generally) do not authenticate to Workspace.
There are no Workspace credentials stored in the dev environment, and the
`gam` binary in a codespace is only useful for refreshing the CSV fixtures
in `fixtures/` if someone manually `gcloud auth login`s and runs a query
in the moment.

The flow for an ad-hoc live audit:

```bash
gh workflow run audit.yml -f dry_run=true     # or omit for a real run
gh run watch
gh run download --name forwarding-audit --dir ./out
```

Tests live against `fixtures/forwarding-sample.csv` and
`fixtures/users-sample.csv`. New fixture data is added by hand when needed.

## Consequences

- **Positive:** Zero Workspace credentials in dev environment. A leaked
  codespace can't leak admin OAuth or the SA.
- **Positive:** One auth model to maintain (WIF in CI), not two parallel
  ones with different gotchas.
- **Negative:** Iteration loop on auth changes is slow — every change
  requires a push + workflow run + log read. The diagnostic step in
  `audit.yml` exists precisely because of this.
- **Negative:** Fixtures can drift from the live CSV shape if GAM7's
  output columns change. Mitigation: parser is column-name tolerant
  (`test/parser.test.ts`).
- **Negative:** New maintainers can't quickly experiment against the real
  workspace without setting up gcloud auth themselves; the documented path
  is "trigger CI."
